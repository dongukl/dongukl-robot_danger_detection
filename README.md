# Scene Risk Perception
**"로봇은 탐지 이후에 무엇을 판단해야 하는가"**

> 기존 객체 탐지 모델은 *무엇이 있는가(What)* 는 답할 수 있지만,  
> *지금 이 상황이 위험한가(Is it dangerous now?)* 는 판단하지 못합니다.  
> 이 프로젝트는 그 **해석 레이어(Interpretation Layer)** 를 직접 설계한 실험입니다.

---

## 한 줄 요약

영상에서 Optical Flow로 움직임 패턴을 추출하고, 전이학습으로 위험도를 분류한 뒤,  
Grad-CAM으로 **"왜 위험하다고 판단했는가"** 의 근거를 시각화합니다.

---

## 파이프라인

```
영상 프레임 입력 (.tif)
        ↓
Optical Flow (Farneback) → Magnitude Map 생성
        ↓
ResNet18 전이학습 (위험 / 정상 이진 분류)
        ↓
Grad-CAM (판단 근거 히트맵 시각화)
        ↓
결과 리포트 (분류 결과 + 근거 이미지 + 실패 케이스 분석)
```

---

## 최종 성능 (테스트셋 기준)

| 지표 | 결과 |
|---|---|
| 정확도 | 98.3% |
| Recall (위험 감지율) | 93.7% |
| Precision (오탐율) | 94.4% |
| F1 Score | 94.1% |

> **Recall을 최우선 지표로 설정한 이유**  
> 위험 상황을 "정상"으로 놓치는 False Negative가  
> 로봇 안전에서 가장 치명적인 실수이기 때문입니다.

## 트러블슈팅 기록

### 문제 1: .tif 파일을 cv2.imread가 읽지 못함

**발견**
프레임 로딩 코드 실행 시 아래 에러 발생.
```
error: (-215:Assertion failed) !ssize.empty() in function 'resize'
```

**원인 분석**
`cv2.imread`가 `.tif` 확장자를 정상적으로 읽지 못해 `None`을 반환.  
이후 `cv2.resize`에 `None`이 들어가면서 에러 발생.

**해결**
`cv2.imread` → `PIL.Image.open()`으로 교체.
```python
# 변경 전
img = cv2.imread(path, cv2.IMREAD_GRAYSCALE)

# 변경 후
img = np.array(Image.open(path).convert('L'))
```

### 문제 2: 클래스 불균형으로 인한 학습 실패

**발견**
10 에폭 학습 후 정확도가 85%에서 멈추고 개선되지 않음.  
손실(Loss)도 거의 변화 없이 평평한 그래프를 보임.

**원인 분석**
데이터셋 구성 확인 결과 심각한 불균형 발견.
```
정상(0): 11,940개 (86%)
이상(1):  1,990개 (14%)
```
모델 예측을 직접 확인한 결과, **모든 샘플을 "정상"으로만 예측**하고 있었음.  
이상 샘플이 적다 보니 전부 정상으로 찍어도 86% 정확도가 나오기 때문에  
모델 입장에서는 학습할 이유가 없는 상태였음.

**해결 방안 검토**

| 방법 | 설명 | 채택 여부 |
|---|---|---|
| 클래스 가중치 부여 | 손실 함수에서 이상 샘플 오답에 더 큰 패널티 | ✅ 채택 |
| 오버샘플링 | 이상 샘플을 복제해서 수를 늘림 | 데이터 다양성 감소 우려 |
| 언더샘플링 | 정상 샘플을 줄임 | 정보 손실 우려 |

**채택 이유**  
데이터를 직접 건드리지 않으면서 불균형을 보정할 수 있고,  
실제 로봇 시스템에서도 데이터보다 손실 함수를 조정하는 방식이 현실적이기 때문.

```python
weight_normal  = total / (2 × 정상 수) ≈ 0.58
weight_anomaly = total / (2 × 이상 수) ≈ 3.50

criterion = nn.CrossEntropyLoss(weight=class_weights)
```

### 문제 3: 가중치 조정만으로는 Recall / Precision 균형을 잡기 어려움

**발견 및 시도별 결과**

| 시도 | 변경 내용 | Recall | Precision | 문제 |
|---|---|---|---|---|
| 1차 | conv1+fc 학습, 가중치 3.5 | 0.87 | 0.23 | 오탐율 너무 높음 |
| 2차 | 가중치 2.0으로 낮춤 | 0.28 | 0.33 | Recall 급락 |
| 3차 | layer4 추가 학습, 가중치 3.5 유지 | **0.937** | **0.944** | ✅ 안정적 |

**원인 분석**  
가중치 숫자만 바꾸는 것은 근본 해결이 아니었음.  
진짜 문제는 **학습하는 레이어가 너무 적어서** 모델이 Optical Flow 패턴을  
충분히 학습하지 못한 것.

```
1차: conv1 + fc만 학습 → 전체의 0.5% 파라미터
3차: conv1 + layer4 + fc 학습 → 전체의 30% 파라미터
```

Optical Flow magnitude 이미지는 일반 RGB 이미지와 완전히 다른 패턴이라  
더 많은 레이어를 재학습시켜야 했음.

**핵심 교훈**  
전이학습에서 "얼마나 많은 레이어를 학습시킬 것인가"는  
데이터의 특성이 사전학습 데이터와 얼마나 다른지에 따라 결정해야 함.

### 문제 4: Grad-CAM 실패 케이스에서 발견한 모델 한계

**발견**  
Grad-CAM으로 실패 케이스(위험인데 정상으로 예측)를 분석한 결과,  
공통 패턴이 발견됨.

**실패 패턴**
```
성공 케이스: 움직임이 크고 한 곳에 집중된 이상 상황 → 정확히 감지
실패 케이스: 움직임이 약하거나 여러 곳에 분산된 이상 상황 → 정상으로 오분류
```

**원인**  
모델이 Optical Flow의 크기(magnitude)에 과도하게 의존하는 경향.  
움직임이 작더라도 방향이 이상한 경우는 감지하지 못함.

**향후 개선 방향**  
Optical Flow의 방향(angle) 정보도 함께 입력으로 사용하면  
이 한계를 극복할 수 있을 것으로 예상.

---

## 한계 및 향후 방향

### 현재 한계
- 움직임이 약하거나 분산된 이상 상황 감지 어려움
- Optical Flow magnitude만 사용 → 방향 정보 미활용
- 야간·저조도 환경에서 성능 저하 예상 (밝기 일정성 가정 위반)
- 카메라 자체가 흔들릴 때 오탐 가능
- 실시간 처리 미검증 (현재 오프라인 분석)

### 향후 확장 방향
- Optical Flow 방향(angle) 정보를 추가 채널로 활용
- LSTM / Transformer로 시계열 패턴 학습
- ROS2 노드로 감싸 실제 로봇에 탑재
- 위험도 점수를 강화학습 보상 함수로 활용
- 궁극적으로: 로봇의 **Safety Layer** — 상위 판단 명령을 하위에서 검토하고 차단하는 안전망
